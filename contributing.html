Contributing

Report bugs

If you find a bug, please create an issue on GitHub, and label it with bug label. Please remember to describe the scenario to reproduce the problem.

Feature requests

You can request new features on the project discussion list. We appreciate new ideas and features, however - remember that the requested feature should be useful for everyone. It may take some time until we start the actual work, but you can speed it up by trying to develop the feature yourself.

Develop Cantiga

Feel free to fork the project on Github. We accept fixes and improvements in the form of pull requests. See the guidelines for developers presented below.

Sending fixes and improvements

We accept the fixes and improvements to Cantiga in the form of GitHub pull requests. Please make sure that your contribution follows some rules:

  • the contributed source code is shared under the terms of GNU General Public License v3, and the files contain the appropriate copyright header,
  • the source code is formatted using PSR-1 coding standards, and use hard TABS for indentation (one tab per indent level),
  • the source code must work correctly under PHP 7.0 and newer versions,
  • the commits were made on a vanilla version of Cantiga, without any customizations. Pull requests made on a customized versions of Cantiga will not be accepted,
  • for database changes, a migration file shall be provided in db/schema directory,
  • the code is consistent with the developer guides published in Cantiga wiki,
  • for bugfixes, the commits must have the issue ID from Github in form 'Fixing #123', where 123 is the issue ID.

The pull request will be reviewed, and the contributor should apply the issues found by reviewers. The project maintainers make the final decision whether to approve or reject the pull requests.

New third-party tools and libraries shall not be introduced without the prior discussion and approval of project maintainers on a discussion list.
